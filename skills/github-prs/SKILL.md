# GitHub PRs

Use this skill to package repository changes into a clear PR-ready artifact.

## Do
- Write concise, reviewer-friendly PR titles
- Summarize intent before listing details
- Highlight testing, risks, and follow-ups
- Make missing validation explicit
- Separate implementation details from reviewer guidance

## Recommended PR body shape
1. Summary
2. What changed
3. Validation
4. Risks / follow-ups
5. Review notes

## Title guidance
- Prefer imperative, specific titles
- Avoid vague titles like `updates` or `fix stuff`
- Keep the title aligned with the primary user-visible change

## Reviewer notes
Call out:
- unusual implementation choices
- areas needing extra scrutiny
- anything intentionally deferred
